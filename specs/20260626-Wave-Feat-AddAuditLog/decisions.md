# 设计决策记录 — 讨论中，无定论

> 以下为讨论过程中的临时记录，尚未最终确认。

## 范围 & 优先级

- 2026-06-26: 审计日志定位为**项目内资产的审计基础设施**，统一收口所有资产类型的关键操作记录（非全局审计、非查询日志）
- 2026-06-26: 用户故事按 P0（核心）→ P1（重要）→ P2（增强）分级，P2 暂无

## 架构 & 技术选型

- 2026-06-26: 使用独立表 `project_audit_log`（meta schema），不在现有资产表内嵌 JSONB（解决 AB 模块的查询痛点）
- 2026-06-26: 审计写入采用 `LogWithFallback` 模式——先尝试写入，失败则兜底仅记录核心字段并打 error log，不阻塞主流程
- 2026-06-26: detail 字段用 JSONB，结构化 `changes: [{field, action, before, after}]` 格式，精确到字段级变更；extra 承载非字段级信息（如 copy 的 source）（来源：PostHog 研究 D-01）

## 边界 & 异常处理

- 2026-06-26: 审计写入失败时静默兜底，主业务流程不受影响、事务不包含审计写入
- 2026-06-26: 单条 detail 序列化后最大 64KB（可配置），超出时优先 LZ4 压缩（复用 `pkg/lib/util/compress.go`）；压缩后仍超则逐字段截断并记录警告（D-09）
- 2026-06-26: 同一资产高频操作每条独立记录，不合并去重
- 2026-06-26: 存量 AB 操作记录全量迁移进新审计表；旧 `details.operation_records` 字段保留不删，后续可废弃。迁移后新操作直接写新表（D-10）

## 数据模型

- 2026-06-26: V1 不做分区，主索引 `(asset_type, asset_id, created_at DESC)`，按需后续补充（D-04/D-06）
- 2026-06-26: action_type 精简为 create/update/delete/copy，状态变更通过 changes[] 字段级 changed 表达（D-03，与 PostHog 理念对齐）
- 2026-06-26: asset_id 允许为 0（存在不关联具体资产的全局操作，用 asset_type 特殊值标识）
- 2026-06-26: 当前不提供跨项目全局查询，由调用方自行跨 schema 查询

## 数据字段

- 2026-06-26: 保留 `operator_name` 字段，写入时快照——用户改名或删除后历史记录仍可追溯操作人（D-11）
- 2026-06-26: 新增 `source` 字段，区分操作来源：web / openapi / mcp / internal（D-13）
- 2026-06-26: 不需要 `operation_group_id`，同一条记录天然是一组操作（D-12）
