## Overview

读取当前 feature 的 `01-spec.md`，生成或更新 **概要设计** `03-plan.md`，严格遵循 `specs/_template/03-plan.md` 的结构。

`plan.md` 的目标是让人能顺着读懂，而不是罗列实现细节：

| Artifact | 回答的问题 |
|---|---|
| `spec.md` | 要做什么 |
| `plan.md` | **总体上怎么做、为什么这样做** |
| `detail.md` | 具体到文件、函数、事务、错误路径怎么落 |

一句话：Plan 是设计文档，不是任务清单。写完后一个不熟悉该 feature 的人应该在 5-10 分钟内理解全貌。

---

## 适用边界

默认产出 `plan.md`。满足以下任一条件时，必须继续进入 `speckit.detail`：

- 涉及数据库 schema / 索引 / migration
- 涉及异步、队列、重试、幂等、分布式补偿
- 涉及跨模块协作或外部依赖
- 涉及权限、安全、审计、缓存一致性
- 涉及 3 个以上核心文件
- 顶层方案已有争议，需把实现边界讲清楚

小改动（局部接口、小型页面、单模块逻辑调整）可以在 plan 后直接进入 `speckit.tasks`，但必须在 plan.md 中说明判断依据。

---

## 写作原则

### 强制约束

1. **先阅读代码再设计** — 不允许凭假设写方案。必须搜索确认现有架构、数据模型、接口签名
2. **必须至少给出 2 种候选方案** — 单一方案等于没选择。没有竞对的方案无法评估优劣
3. **领域模型优先** — 先定义实体和通用语言，再描述技术方案。实体不清则方案必乱
4. **每个 trade-off 必须直说** — 不允许包装成"都很好"。读者需要知道你在什么条件下做了取舍
5. **Mermaid 图是必须的，不是可选的** — 无图则不能说明架构，不允许纯文字段落
6. **方案必须被"拷问"** — 列出至少 5 个最可能被挑战的设计决策

### 常见失败模式

| 失败模式 | 表现 | 预防 |
|---|---|---|
| 假设驱动 | 写着写着发现现有代码不支持 | Step 1 必须先搜索代码确认 |
| 单一方案 | 只写了一个做法，读者无法判断好坏 | Step 2 必须多方案对比 |
| 术语混乱 | A 和 B 用了不同词汇描述同一概念 | Step 1 先提取通用语言 |
| 只写好处 | 方案看起来很美但风险和暗坑没写 | Step 4 必须写出 5 个挑战 |
| 没有地图 | 读者看完不知道改了哪些地方 | 必须包含模块影响矩阵 |

## Steps

### Step 1 — 分析准备

执行以下任务，每项必须完成：

#### 1.1 读取 artifacts

- [x] 读取 `01-spec.md`，理解用户故事、验收场景、边界情况、非功能需求
- [x] 读取 `02-decisions.md`、`HABITS.md`，确认项目约束和既有决策
- [x] 阅读 `02-decisions.md` 中已有的决策记录，避免重复讨论已定事项
- **设计过程中产生新决策时，即时追加到 `02-decisions.md**（参见 CLAUDE.md 「决策记录纪律」）

#### 1.2 代码探索（必须做）

针对 spec 中提到的核心实体、模块、接口，在代码库中进行搜索确认：

```bash
# 搜索示例（根据实际项目调整语言和路径）
grep -r "EntityName" src/ --include="*.go" -l
grep -r "ExistingInterface" src/ --include="*.go" -l
# 查看当前数据模型
find src/db/migrations -name "*.sql" | sort
```

记录搜索结果：**哪些假设被验证、哪些与预期不符**。

#### 1.3 领域建模

从 spec 中提取核心实体和关系：

- 识别实体集合（名词提取法）
- 确定实体之间的关系（一对多 / 聚合 / 组合）
- 定义限界上下文（如涉及多个领域）
- 建立通用语言表（术语 + 定义 + 避免混淆的别名）

#### 1.4 评估变更规模

| 规模 | 判断标准 | 下一步建议 |
|---|---|---|
| 小 | 局部接口/页面/单模块，1-2 文件 | 可跳过 detail |
| 中 | 跨 2-3 模块，有数据模型变更 | 建议 detail |
| 大 | 跨 3+ 模块，涉及外部依赖/异步/安全 | 必须 detail |

---

### Step 2 — 设计候选方案

#### 2.1 生成多个方案

构思至少 2 种（最好 3 种）**在关键维度上不同**的方案。不要只在细节上微调——方案间的差异应体现在架构层面：

- **方案 A**：最直接、最小改动（基准方案）
- **方案 B**：在某个关键约束上优化（如性能/扩展性/运维成本）
- **方案 C（可选）**：激进方案，探索完全不同的技术路径

对每个方案，评估以下维度：

| 维度 | 说明 |
|---|---|
| 核心思路 | 一句话描述 |
| 架构方式 | 模块/层/组件如何组织 |
| 接口复杂度 | 对外暴露的方法数和参数复杂度 |
| 实现工作量 | 相对估算（人天） |
| 扩展性 | 对未来的 2-3 个扩展场景的支持程度 |
| 运维成本 | 部署、监控、排障的难度 |
| 性能影响 | 延迟、吞吐、资源消耗 |
| 可测试性 | 单元/集成测试难度 |
| 隐藏复杂度 | 表面简单但实际易踩坑的地方 |

#### 2.2 评估模块深度

对每个方案涉及的模块，用 codebase-design 的"深度"概念评估：

| 模块 | 当前深度 | 方案下的目标深度 | 说明 |
|---|---|---|---|
| {{Module}} | 浅/深 | 保持/加深/变浅 | 接口大小 vs 实现复杂度 |
| {{Module}} | 浅/深 | 保持/加深/变浅 | |

**设计原则**：Interface 应小而深（少量方法 + 大量实现），避免大而浅（大量方法 + 薄实现）。

#### 2.3 推荐方案

选择推荐方案并给出理由：

1. {{理由 1：最匹配核心约束}}
2. {{理由 2：性价比最优}}
3. {{理由 3：避免的关键风险}}

明确列出不选其他方案的原因，以及**什么条件下需要重新考虑**。

---

### Step 3 — 编写 `03-plan.md`

严格按 `specs/_template/03-plan.md` 的结构输出。产出文件统一命名为 `03-plan.md`（单方案）或 `03-plan-{后缀}.md`（多方案，如 `03-plan-pg.md`、`03-plan-doris.md`）。特别关注以下章节的质量：

#### 3.1 背景与目标（第 1 章）

- 当前问题必须可量化（用户痛点的具体表现）
- 成功标准必须是可验证的（不是"提升性能"而是"P50 延迟 < 200ms"）
- Out-of-scope 必须写"故意不做"的理由，防止后续 scope creep

#### 3.2 领域模型（第 2 章）

- 实体表必须填写完整，不空字段
- 通用语言表特别重要——多个术语指同一概念是设计混乱的信号
- 如果实体关系复杂（超过 3 个实体），必须配关系图

#### 3.3 方案设计（第 4 章）

- 候选方案对比表必须逐行填写，不允许空行
- 选择理由必须与约束条件对应（不要出了一个约束对应的理由）
- 隐藏复杂度是最有价值但最容易被忽略的维度——必须刻意思考

#### 3.4 架构总览（第 5 章）

- 架构图必须标注新增/修改/不变的组件（用颜色或标签）
- 时序图必须展示关键调用链路的全路径
- 模块深度分析如果涉及"保持深度"的模块可简写，但涉及"改变深度"的必须详述

#### 3.5 模块影响矩阵（第 6 章）

- 变更类型精确：新增 / 修改 / 删除 / 仅引用（接口引用不算修改）
- 影响程度不能全写"高"或全写"低"——区分度是判断设计理解的标尺
- 风险等级明确评估依据

#### 3.6 设计拷问（第 8 章）

这是最容易被跳过的章节，但它是最有价值的部分。必须回答以下类型的问题：

- "为什么不用 XXX 方式？"
- "如果数据量增长 10 倍会怎样？"
- "如果这个模块的负责人离职了，新人能维护吗？"
- "这个设计中最脆弱的环节是什么？"
- "回滚方案是什么？"

#### 3.7 落地顺序（第 9 章）

- Phase 1 必须是可独立交付的最小可行
- 每个 Phase 的依赖关系必须明确

---

### Step 4 — 质量自查

逐项检查 `03-plan.md` 末尾的 Quality Gates：

- [ ] 背景与目标清晰：问题定义、成功标准、out-of-scope 完整
- [ ] 领域模型完整：核心实体、关系、通用语言已定义
- [ ] 代码探索确认：已通过代码搜索确认现有架构与假设一致
- [ ] 多方案对比：至少 2 种候选方案，有对比表格和选择理由
- [ ] 架构图完整：包含 Mermaid 系统架构图和核心数据流图
- [ ] 模块影响明确：每个受影响模块的变更类型和风险等级已标注
- [ ] 设计已拷问：至少 5 个潜在挑战已记录和回应
- [ ] 风险已分析：性能、可靠性、安全、复杂度均有评估
- [ ] 落地顺序可行：Phase 划分合理，依赖关系清楚
- [ ] 下一阶段有明确建议：tasks 或 detail，含判断依据

任一 gate 未通过则 plan.md 视为草稿，不得 handoff。

---

### Step 5 — handoff 决策

根据 Step 1.4 的评估：

- **中大变更** → handoff 到 `speckit.detail`
- **小变更** → handoff 到 `speckit.tasks`
- **复杂方案** → 建议用户先运行 `/grill-me` 或 `/gstack-plan-eng-review` 拷打方案

## 质量门禁

> 以下为 Dev 阶段关口使用，但 Plan 阶段必须确认前置条件满足。

- [ ] 数据模型变更已明确（不要求细节，但必须有方向）
- [ ] 外部依赖变更已识别（新增/升级/废弃）
- [ ] 安全影响已初步评估（是否需要审计/权限变更）
